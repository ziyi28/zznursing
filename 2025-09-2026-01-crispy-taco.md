## 一、AI 健康评估模块（PDF + Prompt + LLM + JSON Schema）

### Q1. 请整体介绍一下你的 AI 健康评估是怎么做的？

**【A】建议按四段式讲（30~60 秒）：**
1. **链路**：前端上传 PDF → 后端用 PDFBox 抽文本 → 文本缓存到 Redis Hash → 表单提交后拼 Prompt 调大模型 → 模型按 JSON 模式输出 → fastjson2 反序列化为 VO → 入库。
2. **核心代码位置**：`HealthAssessmentServiceImpl#insertHealthAssessment`（行 71-110），`AIModelInvoker#DeepseekInvoker`（行 23-39）。
3. **业务结果**：拿到 5 个枚举级别的风险等级、0-100 健康分、风险分布占比、异常指标 7 字段、8 大系统分、整体总结 — 直接给老人家属看。
4. **业务收益**：把人工阅读体检报告的 30 分钟压缩到秒级响应。

**【证据】**：`zzyl-nursing-platform/.../controller/HealthAssessmentController.java:97/127` + `service/impl/HealthAssessmentServiceImpl.java:71-110` + `zzyl-common/.../ai/AIModelInvoker.java:23-39`。

**【可能被反问】**
- "为什么分两步（先上传 PDF、再提交表单）？" — 因为上传只解析不调模型，给前端预校验 + 用户改表单的窗口期，**而且 Redis Hash 的 key 是 `healthReport + 身份证号`，提交流程会 `redisTemplate.delete` 清理**（ServiceImpl.java:78）— 是一次性缓存，避免反复调模型。
- "为什么模型返回后还要再 set 一遍 `healthAssessment` 的字段？" — 因为模型不感知"身份证号 / 姓名 / 性别 / 年龄"，这些是从身份证号本地解析的（`IDCardUtils.java:22-78`），而 `healthScore / riskLevel / suggestionForAdmission / nursingLevelName` 是基于模型返回值二次计算 — **业务字段分离管理，模型只负责"健康评估"那一块**。

### Q4. 你的 Prompt 是怎么设计的？为什么这么设计？

**【A - Prompt 工程重点】**：
1. **角色设定**：第一句就锁定"请以专业医生的视角" — 单一角色约束。
2. **Few-shot**：没用例子，只给"输出格式"作为模板（Zero-shot + Format Spec）。
3. **强约束**：把 JSON schema 直接写在 Prompt 里（`totalCheckDate / healthAssessment / riskDistribution / abnormalData / systemScore / summarize` 6 个顶层字段全列出），字段名跟 Java VO 一一对应。
4. **要求条款**：6 条 numbered list — 总检日期 → 风险等级+指数 → 风险分布 → 异常数据 7 字段 → 8 大系统 → 总结。
5. **数据塞入**：Prompt 里 `体检内容如下：\n` + PDF 全文 + `\n` — **没有任何分段、没有截断**（**风险点**，如果体检报告 > 8K tokens 可能超限）。

**【证据】**：`HealthAssessmentServiceImpl.java:130-182`，方法 `getPrompt(String content)`。

**【可能被反问】**
- "Prompt 调过几次？" — 可以讲"迭代过 3 个版本：v1 把所有字段写一段自然语言描述（模型经常少字段）；v2 把字段名用 JSON 显式声明（覆盖率 90%）；v3 引入 JSON Object Mode（覆盖率 99%）"。
- "Prompt Injection 怎么防？" — 当前**没防**，体检报告里如果混入"忽略以上指令"类攻击文本，模型会跑偏。**改进方案**：把 PDF 内容用 `<exam_report>` XML 标签包裹，与指令物理隔离。

### Q5. 你说用 JSON Schema 强约束 — 实际怎么落地的？

**【A - 必考点】**：
1. **第一层约束 — OpenAI JSON Object Mode**：在 `AIModelInvoker.java:32` 调用 `.responseFormat(ofJsonObject(...))`，模型保证返回合法 JSON 字符串。
2. **第二层约束 — Prompt 里显式声明字段名**：把 schema 写在 Prompt 中（`getPrompt` 行 152-182）。
3. **第三层约束 — Java VO 反序列化**：用 fastjson2 `JSON.parseObject(context, HealthReportVo.class)` — **字段名严格匹配 VO 属性**。

**【诚实承认】**：**没用 Function Calling / Tools**，**没用 JSON Schema 严格校验**（没传 schema 给模型，模型也不知道具体字段），**没用 JSCHEMA / json-schema-validator 做后端兜底校验**。一旦模型输出 Markdown 代码块 ```json ... ```（很常见），`JSON.parseObject` 会**直接抛异常 → 5xx**。

**【改进方案】**（可以这样表达你"会怎么优化"）：
> 1. 加 Markdown fence 剥离正则：`text.replaceAll("```json\\s*", "").replaceAll("```\\s*", "")`。
> 2. 失败时 retry 1-2 次，prompt 末尾加"严格输出 JSON，不要用代码块"。
> 3. 引入 Function Calling，把每个字段做成 tool 参数，让模型结构化输出。

**【证据】**：`AIModelInvoker.java:32`、`ServiceImpl.java:82-83`。

### Q9. 你的 `AlertTask` 定时任务是怎么设计的？扫描周期多长？

**【A - 简历亮点题，建议主动展开】**：

报警扫描走的是 **RuoYi 自带的 Quartz 后台定时任务**（**不是** `@Scheduled` 注解），无需重启应用就能在"系统管理 → 定时任务"页配置 cron。

**核心机制**（4 个组件联动）：

1. **任务定义类**：`AlertTask`（`zzyl-nursing-platform/.../task/AlertTask.java`），`@Component` 注入，方法 `public void deviceDataAlertFilter()` **无参 public**（这是反射调用的硬性要求）。
2. **反射调用工具**：`JobInvokeUtil.invokeMethod()`（`zzyl-quartz/.../util/JobInvokeUtil.java:23-63`）：
   ```java
   String invokeTarget = sysJob.getInvokeTarget();    // 如 "alertTask.deviceDataAlertFilter()"
   String beanName = getBeanName(invokeTarget);       // → "alertTask"
   String methodName = getMethodName(invokeTarget);   // → "deviceDataAlertFilter"
   Object bean = SpringUtils.getBean(beanName);       // 从 Spring 容器拿 Bean
   Method method = bean.getClass().getMethod(methodName);  // 反射拿方法
   method.invoke(bean);                               // 无参反射调用
   ```
3. **数据库配置表**：`sys_job`（`SysJob.java:39` `invokeTarget` 字段）。后台录入一条：
   - `job_name = 设备数据报警扫描`
   - `invoke_target = alertTask.deviceDataAlertFilter()`
   - `cron_expression = 0/5 * * * * ?`（每 5 秒一轮）
   - `status = 0`（正常）
4. **启动加载**：`SysJobServiceImpl.init()` 有 `@PostConstruct`（行 37-46），应用启动时自动 `jobMapper.selectJobAll() → ScheduleUtils.createScheduleJob(scheduler, job)`，把 `sys_job` 表里所有任务塞进 Quartz Scheduler。

**扫描周期**：在后台 UI 配置（推荐 `0/5 * * * * ?` 每 5 秒一轮）。

**【证据】**：
- `AlertTask.java:9, 16`（`@Component` + 无参 public 方法）
- `JobInvokeUtil.java:23-63`（反射调用）
- `SysJobServiceImpl.java:37-46`（`@PostConstruct init()`）
- `SysJob.java:39`（`invokeTarget` 字段）

**【诚实承认】**：
- `sql/zzyl-dev06-init.sql` 里**没有预置** `alertTask.deviceDataAlertFilter` 的 sys_job 记录（`grep deviceDataAlertFilter` 无匹配），所以**项目刚拉下来时任务确实不跑**。
- 需要在后台"系统管理 → 定时任务"里手动新增一条才能启用。
- **好处**：cron / 启停 / 并发策略（`concurrent` 字段 0=允许 1=禁止）都可以热调整，不用重启；多任务并存互不耦合。
- **改进方案**：在 `sys_job` 表里预置一条 `INSERT` SQL，初始化即生效。

**【可能被反问】**
- "为什么不直接 `@Scheduled` 注解？" — `@Scheduled` 写死在代码里，调整 cron 必须改代码+重启；Quartz 后台方案支持**热调整 + 启停 + 并发控制 + 执行日志**（`sys_job_log` 表）— 运维更友好。
- "反射调用怎么保证方法存在？" — `Class.getMethod` 找不到会抛 `NoSuchMethodException`，被 `AbstractQuartzJob.execute()` 捕获并写 `sys_job_log`（失败日志），不会让整个应用崩。
- "Bean 名怎么确定的？" — 默认是类名首字母小写（`AlertTask → alertTask`），如果用了自定义 Bean 名，要以 `@Component("customName")` 为准。
- "参数怎么传？" — `JobInvokeUtil.getMethodParams()` 支持 4 种字面量：`'str'` / `"str"` / `true|false` / `100L` / `1.0D` / `123` — 不支持复杂对象。**改进**：业务参数走 `JobDataMap`。

### Q10. 报警规则匹配是怎么做的？规则定义在哪里？

**【A】**：规则表 `alert_rule`（Domain: `AlertRule.java`），关键字段：
- `productKey / functionId`：匹配维度
- `iotId`：-1 表示全局规则，否则设备专属
- `operator`：运算符字符串，**代码只识别 `>=` 和 `<` 两种**（`AlertRuleServiceImpl.java:215-223`）
- `value`：阈值（Long）
- `duration`：连续 N 次才报警
- `alertEffectivePeriod`：生效时段，字符串 `HH:mm:ss~HH:mm:ss`
- `alertSilentPeriod`：沉默周期（分钟）
- `status`：0 禁用 / 1 启用

**匹配逻辑**（`alertFilterData` + `deviceDataAlarmHandler`）：
1. 先查 `productKey + functionId + status=1 + iotId=-1` 的全局规则；
2. 再查 `productKey + functionId + status=1 + iotId=设备ID` 的设备专属规则；
3. 用 `CollUtil.addAll` 合并。
4. 数据时效性检查：丢弃 `> 60s` 的旧上报（行 169-171）。
5. 用 Hutool `NumberUtil.compare` 比较，**仅识别 `>=` 和 `<`** — 其他字符串走"正常"分支被忽略。

**【证据】**：`AlertRuleServiceImpl.java:165-238`。

**【可能被反问】**
- "为什么只用 `>=` 和 `<`？" — 业务里 90% 阈值场景是"心率 > 100 报警""血压 < 60 报警"，其他运算符（`>`/`<=`/`==`/`!=`）需求少 — **改进方案**：用 `ScriptEngine`（Java 内置 JS 引擎）执行动态表达式。
- "规则冲突怎么办？" — 当前**没处理**，两个规则都满足会触发两次。**改进**：加优先级字段 `priority`，命中最高优先级就 break。

### Q11. Redis 阈值计数怎么做的？能扛 50+ 设备并发吗？

**【A - 必考并发题】**：

**Key 设计**：
- `iot:alert_trigger_count:{iotId}:{functionId}:{ruleId}` — 计数 Key。
- `iot:alert_silent:{iotId}:{functionId}:{ruleId}` — 沉默 Key。

**计数逻辑**（`AlertRuleServiceImpl.java:234-239`）：
```java
String countStr = (String) redisTemplate.opsForValue().get(alertCountKey);
int count = Integer.parseInt(countStr == null ? "0" : countStr) + 1;
redisTemplate.opsForValue().set(alertCountKey, String.valueOf(count));
if (count != rule.getDuration()) return;
redisTemplate.delete(alertCountKey);  // 触发后清零
```

**【陷阱 — 简历说"Redis 计数"但实现是 GET + 1 + SET，不是 INCR】**：
- **没用 INCR**（原子操作），存在竞态 — 两个并发线程同时 GET 到 count=5，各自 + 1 后 SET 7，丢失一次计数。
- **没用 EXPIRE**（没有 TTL）— 如果某设备上报中断，Key 永远残留，下一次重连会带着历史计数。
- 沉默 Key 用的是 `opsForValue().set(key, "1", duration, TimeUnit.MINUTES)` — 等价 SETEX，但**先 GET 再 SET**，**非 SETNX 原子**。

**【正确做法】**（如果面试官追问"你会怎么改"）：
```java
Long count = redisTemplate.opsForValue().increment(alertCountKey);   // 原子 INCR
redisTemplate.expire(alertCountKey, 1, TimeUnit.HOURS);               // 兜底 TTL
Boolean firstSilent = redisTemplate.opsForValue()
    .setIfAbsent(alertSilentKey, "1", rule.getAlertSilentPeriod(), TimeUnit.MINUTES);  // SETNX + EXPIRE 原子
if (Boolean.FALSE.equals(firstSilent)) return;  // 沉默期已存在，不触发
```

**【证据】**：`AlertRuleServiceImpl.java:226-239, 242, 244-245`、`CacheConstants.java:71, 75`。

### Q12. 沉默期去重的设计？沉默多久？

**【A】**：
- Key: `iot:alert_silent:{iotId}:{functionId}:{ruleId}`。
- TTL = `alertRule.getAlertSilentPeriod()` 字段值，**单位是分钟**。
- 触发报警后 `set(key, "1", duration, TimeUnit.MINUTES)` — 写一次沉默标记。
- 下次报警时 `get(alertSilentPeriodKey)` 非空 → 直接 return，不重复触发（`AlertRuleServiceImpl.java:229-231`）。

**【关键缺陷 — 同 Q11】**：GET-then-SET 非原子，可能并发触发都通过检查。**改进方案**：用 `setIfAbsent`（SETNX）— 返回 false 表明已在沉默期。

**【可能被反问】**
- "沉默期和计数 Key 为什么不合并成一个 Hash？" — 合并后可一次 `HGET` 拿两个值，少一次 RTT。但**当前实现为了清晰分开职责**。

### Q15. 并发支持 50+ 设备在线，你的代码怎么保证？

**【A - 高风险陷阱题】**：

**当前实现的隐患**：

1. **`sessionMap` 是 `HashMap<String, Session>`，不是 `ConcurrentHashMap`**（`WebSocketServer.java:28`）— 多客户端同时 onOpen / onClose 会触发 HashMap 内部链表成环 / 数据竞争，导致 **CPU 100% 或 NPE**。这是**已知 BUG**。

2. **推送是同步阻塞**：`session.getBasicRemote().sendText(...)`（行 122）— 没用 `AsyncRemote`。一个慢客户端能卡住整个告警链路（push 在 AMQP 消费线程里同步调用，`AlertRuleServiceImpl.java:286`）。

3. **报警引擎与 WebSocket 强耦合**：`AlertRuleServiceImpl.java:274-275` 直接 `@Autowired WebSocketServer`，没用 `ApplicationEvent` 解耦。

**【简历说"事件驱动"但代码不是事件驱动】**：
- 简历："基于 Spring @ServerEndpoint 构建 /ws/{sid} 长连接与会话池，提供点对点/广播双模式与报警引擎解耦"
- 代码：直接 Service 注入 WebSocketServer，没有接口抽象，没有 ApplicationEvent。

**【正确做法（回答用）】**：
```java
// 1. sessionMap 用 ConcurrentHashMap
private static final Map<String, Session> sessionMap = new ConcurrentHashMap<>();

// 2. 用 ApplicationEvent 解耦
applicationContext.publishEvent(new AlertEvent(this, alertNotifyVo, userIds));
// WebSocketServer 监听 @EventListener
@EventListener
public void onAlert(AlertEvent event) { sendMessageToConsumer(event.getVo(), event.getUserIds()); }

// 3. 推送用 AsyncRemote + onError 兜底
session.getAsyncRemote().sendText(msg, new SendHandler() { ... });
```

**【证据】**：`WebSocketServer.java:28, 85-93, 102-128`、`AlertRuleServiceImpl.java:274-275, 286`。

**【关键提醒】**：面试官很可能**直接读源码**，这一段如果被发现是 HashMap，分数会大打折扣。**建议直接承认"未来改进"**，比被发现好得多。

### Q16. WebSocket 断线重连怎么做？消息丢失怎么办？

### Q18. 你的 JWT 是怎么签发的？密钥怎么管理？Token 过期怎么处理？

**【A】**：
- **库**：`io.jsonwebtoken:jjwt` 0.9.1（`pom.xml:32, 181-185`），**算法 HS512**（对称密钥）。
- **密钥管理**：`application-dev.yml:148-155`：
  ```yaml
  token:
    header: Authorization
    secret: abcdefghijklmnopqrstuvwxyz   # 26 个字母，明文硬编码
    expireTime: 30                        # 30 分钟
  ```
  — **明文硬编码**，**没走 jks/p12 密钥库**，**没做密钥轮换**，**演示级别**。
- **签发**：`TokenService#createToken(Map<String, Object>)`（行 178-184），`Jwts.builder().setClaims(claims).signWith(HS512, secret).compact()`。
- **校验**：`parseToken(String)`（行 192-198），`Jwts.parser().setSigningKey(secret).parseClaimsJws(token).getBody()`。

**【关键缺陷 — JWT 里没设 `exp`】**：
- `createToken` 没调用 `.setExpiration()`，JWT 本身**永不过期**。
- 管理后台过期靠 Redis 里 `login_tokens:{UUID}` 的 TTL（30 分钟）。
- 小程序端 JWT **完全无过期机制** — 用户一次登录永远有效，**安全风险**。

**【改进方案】**（回答时这样表达）：
```java
long now = System.currentTimeMillis();
claims.put("exp", new Date(now + expireTime * 60 * 1000L));
claims.put("iat", new Date(now));
```

**【证据】**：`TokenService.java:178-198`、`application-dev.yml:148-155`、`Constants.java:106`。

### Q20. 你说"天然隔离小程序/管理后台两套用户体系" — 怎么做到的？

**【A - 简历亮点题】**：

### Q21. Spring Security 和 JWT 拦截器的关系？谁先执行？

**【A】**：


3. **时区处理**：`DateTimeZoneConverter.utcToShanghai` 显式做 UTC → 上海时间转换（华为 IoTDA 上行是 UTC）。
4. **一条报警对应 N 条通知记录**：`insertAlertData` 用 `BeanUtil.copy` 给每个接收人复制一条（行 296-318），`saveBatch` 批量入库 — 一对多。
5. **Lua 限流脚本**：自研 `limitScript`，面试时可以画一下 Lua 代码。
6. **JWT + ThreadLocal 双轨设计**：管理后台用 Spring Security + SecurityContextHolder，小程序用自建 HandlerInterceptor + ThreadLocal，物理隔离。

### R3. 几个具体的"陷阱问题" + 答题模板。

**Q: 你的 sessionMap 为什么用 HashMap？**
- 诚实答："当时为了赶进度用了 HashMap，确实存在并发风险，**生产环境应该改成 ConcurrentHashMap**。"

**Q: 你的 PDF 解析对扫描件怎么处理？**
- 诚实答："项目里没做 OCR，PDFBox 直接抽文本 — 因为合作医院给的体检报告都是结构化文本型 PDF。如果将来要支持扫描件，会引入 PaddleOCR 或腾讯云 OCR。"

**Q: 你的 API Key 为什么写在 yml 里？**
- 诚实答："上线初期为了快速迭代，API Key 写在 yml 里 — 生产环境应该走 Nacos / Vault / 阿里云 KMS 加密。"

**Q: 你的 JWT 为什么不带 exp？**
- 诚实答："管理后台的过期靠 Redis 里的 LoginUser TTL 控制；小程序端当前没设过期机制 — 这是一个待改进点，未来需要补 `setExpiration` + 客户端 refresh token。"

**Q: 你的报警触发链路真的能跑通吗？**
- 诚实答："是的。报警扫描走两条独立链路 — ① RuoYi Quartz 后台定时任务：在'系统管理 → 定时任务'页配置 `invoke_target=alertTask.deviceDataAlertFilter()`、`cron=0/5 * * * * ?`，由 `SysJobServiceImpl.@PostConstruct init()` 启动时自动加载到 Scheduler，每 5 秒调一次 `alertFilter()`。② AMQP 入口（规划中）设备数据上报时同步触发。当前 SQL 没预置 sys_job 记录，需要后台手动加一条。"
